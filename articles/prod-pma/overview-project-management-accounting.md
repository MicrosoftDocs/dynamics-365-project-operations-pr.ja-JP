# <a name="project-management-and-accounting-overview"></a>プロジェクト管理および会計の概要

[!include [banner](../includes/banner.md)]

プロジェクト管理および会計機能は、サービスの提供、製品の製造、または結果の達成のために複数の業種で使用できます。  

プロジェクトとは、サービスの提供、製品の製造、または結果の達成を目的として設計された活動のグループです。 プロジェクトは、リソースを消費し、売上または資産の形式で財務結果を生成します。

## <a name="projects-across-industries"></a>業界全体のプロジェクト
プロジェクト管理と会計の機能は、次の図に示すように、複数の業種で使用できます。

[![業界全体のプロジェクト](./media/projects-accross-industries.jpg)](./media/projects-accross-industries.jpg) 

コール センターでは、1 つのチケットを使用して、通話を解決するために必要な一連のアクションを記述できます。 コンサルティング会社 (経営またはテクニカル コンサルティング組織や広告代理店など) は、その活動をプロジェクトとして参照します。 マーケティングでは、キャンペーンは、果たす必要がある一連の作業を表します。 プロジェクトベースの製造では、製造オーダーとは、完成品を製造するために実行する必要があるさまざまな作業と関連しています。 どのような名前を使用する場合でも、これらのプロジェクトにはリソース、スケジュール、費用が関係するため、プロジェクト管理と会計機能を使用すると、これらのプロジェクトの計画、実行、分析に役立ちます。

## <a name="project-phases"></a>プロジェクト フェーズ

## <a name="initiate-the-project"></a>プロジェクトの開始
プロジェクトの開始時に、いくつかの主要プロセスが発生します。 プロジェクト見積を使用して、見積もられた労働力、費用、および資料を顧客に伝えることができます。 プロジェクト契約には、請求条件、制限、および合意した内容を記録できます。 WBS (作業分解構造) を使用して、作業を計画し、見積もることができます。 予測と予算を設定して、プロジェクトの実行をガイドできます。 次の図は、プロジェクトの構造を示しています。[![プロジェクト構造](./media/project-structure1.jpg)](./media/project-structure1.jpg)  

### <a name="create-project-quotations"></a>プロジェクト見積の作成

プロジェクトの最初の営業フェーズでは、プロジェクトの見積により、顧客に非バインドのオファーを提供できます。 見積には、見積もられた品目やサービス、基本的な連絡先情報、特別な売買契約や値引き、税金や割増金が含まれる場合があります。

また、組織と顧客の間のプロジェクト見積トランザクションの信用保証状を発行することもできます。 プロジェクト見積を作成したら、顧客に対して信用保証状を作成し、銀行に送信できます。 銀行が要求を承認した後、信用保証状が顧客に発行されます。 

詳細については、[プロジェクト見積](project-quotations.md) を参照してください。

### <a name="create-project-contracts"></a>プロジェクト契約の作成

プロジェクトを完了するために顧客または他の資金調達ソースとの契約を結ぶ場合、最初にプロジェクト契約を作成する必要があります。 次に、プロジェクトを作成するときに、対応する契約に割り当てる必要があります。 プロジェクト契約に対して作成するプロジェクトのタイプによって、プロジェクト顧客への請求に使用される方法が決まります。 プロジェクト契約と関連プロジェクトは変更できますが、プロジェクト タイプは変更できません。 プロジェクト タイプの詳細については、「プロジェクトの作成」セクションを参照してください。

プロジェクト契約の詳細については、[プロジェクト契約](project-contracts.md) を参照してください。

### <a name="create-work-breakdown-structures"></a>WBS (作業分解構造) の作成

WBS の詳細の程度は、見積もりに必要な正確性のレベルと、それらの見積もりに対して必要な追跡のレベルに依存します。 スケジュールまたは原価のずれに対する許容範囲が非常に低いプロジェクトでは、通常、より詳細な WBS と WBS に対する作業の進捗とコストの入念な追跡が必要です。 

詳細については、[WBS (作業分解構造) の概要](work-breakdown-structures.md) を参照してください。

### <a name="create-project-forecasts-and-budgets"></a>プロジェクトの予測と予算の作成

組織に運用上の観点があり、特定のトランザクションから派生している売上およびコストに重点を置いている場合は、予測を使用できます。 ただし、組織が財務金額に重点を置いている場合は、予算作成を使用できます。 それぞれの方法に利点があります。 詳細については、[プロジェクトの予測と予算](project-forecasts-budgets.md) を参照してください。

### <a name="assign-workers-categories-and-resources"></a>作業者、カテゴリ、リソースを割り当てる

プロジェクトの要件とスケジュール、または作業者のスキルと可用性に基づいて、作業者リソースをスケジュールできます。 リソース スケジュール機能を使用することにより、組織の作業者を効率的かつ効果的に展開できます。 プロジェクトで作業できる最も適任の作業者をすばやく見つけることができます。 また、プロジェクトの過程でこれらの作業者がどのように効果的に起用されるかを簡単に確認できます。 

リソース スケジュール機能を使用する方法を次に示します。

-   教育、スキル、資格、プロジェクトの経験など、作業者の属性に関する情報を使用して、作業者をプロジェクトの要件に合わせます。
-   作業者のカレンダーと空き時間に関する情報を使用して、作業者のスケジュールをプロジェクト カレンダーに合わせます。
-   各作業者のキャパシティを確認し、そのキャパシティの使用方法を決定します。 たとえば、作業者が十分に活用されていない場合、その作業者の可用性と属性に適合するプロジェクトに割り当てることができます。
-   作業者の可用性を確認して、作業者の割り当てとカレンダーが競合していないことを確認します。
-   概要ビュー (部門別、作業者別など) または詳細ビュー (部門内の作業者別、または各作業者の週次詳細など) で、作業者の稼働率に関する情報を確認します。
-   日、週、月などのさまざまな時間単位のリソース割り当てを変更して、作業者の起用方法を最適化します。

## <a name="execute-the-project"></a>プロジェクトの実行
プロジェクトの実行中、チーム メンバーまたはマネージャーは、タイムシート、経費精算書、およびその他のビジネス ドキュメントを使用して、発生した作業と経費を記録します。 プロジェクト マネージャーには、プロジェクトの予算額の消費状況を監視できるツールがあります。 プロジェクト マネージャーは、発注書やその他のビジネス ドキュメントを使用して、プロジェクトの資料を受注、ピック、調達をすることもできます。 請求書が作成および承認されるため、信仰中の作業に対して顧客に請求できます。 最後に、売上はこのプロセス中に認識され、組織の財務に影響を与えます。

### <a name="manage-work-breakdown-structures"></a>WBS (作業分解構造) の管理

WBS (作業分解構造) は、プロジェクトに対して完了する作業の説明です。 WBS はタスクの階層です。 これは、各タスクの作業だけでなく、タスクのサイズ、コスト、および期間も表します。 

詳細については、[WBS (作業分解構造) の概要](work-breakdown-structures.md) を参照してください。

### <a name="manage-project-forecasts-and-budgets"></a>プロジェクトの予測と予算の管理

プロジェクトを管理および制御するために、プロジェクトの予測とプロジェクトの予算という 2 つの方法があります。 組織に運用上の観点があり、特定のトランザクションから派生している売上およびコストに重点を置いている場合は、予測を使用できます。 ただし、組織が財務金額に重点を置いている場合は、予算作成を使用できます。

詳細については、[プロジェクトの予測と予算](project-forecasts-budgets.md) を参照してください。

### <a name="create-production-orders"></a>製造オーダーの作成

プロジェクト関連の製造オーダーは、完成品メソッドまたは消費済み品目メソッドのいずれかを使用して、受注または品目要件にリンクできます。 さらに、製造オーダーが手動で作成された場合、製造オーダーと受注または品目要件の間にリンクはありません (受注へのリンクはありません)。 ただし、製造オーダーが受注または品目要件を満たすために自動的に作成された場合、製造オーダーと受注または品目要件の間にはリンクがあります (受注へのリンク)。 

これらの要因の組み合わせに基づいて、次のいずれかの方法を使用します。

- **完成品/受注へのリンク** – プロジェクトを受注または品目要件にリンクします。 この方法を使用すると、販売注文の請求時または品目要求に対する梱包明細の更新時に、実際のプロジェクト コストが転記されます。 コストは完了品目として転記されます。
- **完了品目/注文へのリンクなし**: 品目の生産サイクルのステータスが **終了** になるまで、実際原価は転記されません。 完了品目のコストは 1 つのトランザクションとして転記されます。
- **消費品目/注文へのリンク**: プロジェクトを品目要求にリンクします。 この方法を使用すると、生産のステータスが **開始済** または終了としてレポートされた場合に実際のプロジェクト コストを表示できます。 コストは、生産に消費される原材料と時間に対する複数のプロジェクト品目トランザクションとして転記されます。 品目要求に対して梱包明細が更新された場合、プロジェクト コストは転記されません。 生産時のプロジェクトの追跡に使用する部品表 (BOM) 階層のレベルを定義することもできます。
- *<strong><em>消費された品目/受注へのリンクなし</em></strong>* – プロジェクトを品目要件にリンクします。 このメソッドを使用すると、運用のステータスが<strong>開始</strong>であるか、終了済みと報告されたときに、実際のプロジェクト コストを表示できます。 コストは、生産に消費される原材料と時間に対する複数のプロジェクト品目トランザクションとして転記されます。 運用のプロジェクトが追跡される BOM 階層のレベルを定義することもできます。

#### <a name="sales-orders-for-projects"></a>プロジェクトの受注

プロジェクト管理と会計では、いくつかの方法で品目の消費を登録できます。 プロジェクトから品目を販売または購入したり、プロジェクトの品目を予約したりできます。 

プロジェクトで消費するために会社の在庫から品目を注文できます。 または、外部ベンダーから品目を購入することもできます。 品目は、時間プロジェクトを除くすべてのタイプのプロジェクトで使用できます。 

品目の受注方法は、品目の受注元によって異なります。

-   会社の在庫から品目を受注するには、品目要件として受注を入力する必要があります。 **品目要件** ページを使用する場合、品目を部分配送として受け取るように要件を設定できます。 したがって、品目が必要になるまで品目の数量の消費を延期できます。
-   外部ベンダーから品目を受注するには、**発注書** ページで発注書として受注を作成する必要があります。

> [!NOTE] 
> プロジェクト関連の受注の梱包明細は、品目が既に梱包対象としてマークされている場合はキャンセルできません。 

次の表で品目の注文方法を示し、品目の消費方法を説明します。

### <a name="process-project-invoices"></a>プロジェクト請求書の処理

どの請求手順を適用するかは、プロジェクト タイプによって決まります。 請求できるのは、時間/実費払と固定価格の 2 つの外部プロジェクト タイプだけです。 時間/実費払プロジェクトおよび固定価格プロジェクトは、常にプロジェクト契約に付随します。 

プロジェクトの顧客請求書を作成する前に、事前請求書または仮発行請求書を作成できます。 仮発行請求書で、プロジェクト請求書に含めるプロジェクト トランザクションを選択できます。 その後、プロジェクト請求書を転記する前に、請求書の詳細を確認して、顧客または他の資金調達ソースに送信できます。 


プロジェクト請求書の処理方法については、[プロジェクト請求](/dynamics365/finance/accounts-payable/project-invoicing) を参照してください。

## <a name="analyze-the-project"></a>プロジェクトの分析
最も基本的なレベルでは、プロジェクトは、コストを記録するトランザクションをグループ化し、これらのコストを総勘定元帳に転記するために使用されます。 

通常、これらのトランザクションは、タイムシート、経費精算書、仕入先請求書、在庫トランザクションなどのビジネス ドキュメントの結果です。 プロジェクトのライフ サイクルは通常、プロジェクトの作業や財務的影響を計画および予想するのに役立つ見積もり、予測、および予算から開始されます。 プロジェクトを分析すると、プロジェクト中に発生したトランザクションだけでなく、見積もりと予測の正確さ、プロジェクト チーム メンバーの稼働率、およびプロジェクトの全体的な成功も評価できます。

### <a name="analyze-cash-flow"></a>キャッシュフローの分析

キャッシュ フローの監視を使用して、プロジェクトの予測キャッシュ フローと実際のキャッシュ フローの両方を確認します。 プロジェクトの進行中にキャッシュ フローを確認することも、完了したプロジェクトのキャッシュ フローを表示することもできます。 

キャッシュ フローを監視することにより、単一のプロジェクトを評価し、レポートを使用して複数のプロジェクトを表示し、プロジェクトのキャッシュ フローを総勘定元帳のキャッシュ フロー予測に転送できます。

#### <a name="cash-inflow-forecasting"></a>現金収支の予測

設定に基づいて、選択したプロジェクトの現金収支を予測できます。 たとえば、プロジェクトの日付が 2012 年 3 月 5 日であり、2012 年 3 月 31 日に請求する場合、次のようにして期限と予想売上支払期日を予測できます。

-   **プロジェクト日:** 2012 年 3 月 5 日。
-   **請求書の日付:** 2012 年 3 月 31 日。 この日付は、請求書の頻度に基づいて決定されます。 この例では、請求頻度を現在の月に設定します。 したがって、3 月に転記されたすべてのトランザクションは、月の最終日に請求されます。
-   **期限:** 2012 年 4 月 14 日。 この日付は、プロジェクトに設定された支払条件に基づいて決定されます。 この例では、14 日間の支払条件を選択しました。 したがって、請求日に 14 日が追加され、2012 年 4 月 14 日の期日になります。
-   **予想売上支払期日:** 2012 年 4 月 27 日。 この日付は、**プロジェクト管理および会計パラメーター** ページの **一般的なバッファー日** フィールド内の日数に、**プロジェクト契約** ページの **個別のバッファー日** フィールド内の日数を加算し、その合計を **期限** フィールドの日数に加算して計算します。 この例では、**一般的なバッファー日** フィールドに **3** を入力し、**個別のバッファー日** フィールドに **10** を入力しました。 したがって、期限に 13 日が追加されたので、予想売上支払期日は 2012 年 4 月 27 日になります。

一般的なバッファー日は、個別のバッファー日を置き換えるか、または個別のバッファー日に追加することができます。

-   一般的なバッファー日を個別のバッファー日の代わりに使用するには、期限から実際の顧客の支払期日までの平均日数を入力します。
-   一般的なバッファー日を個別のバッファー日に追加するには、**一般的なバッファー日** フィールドに、顧客が支払を送信する日と組織が支払を受け取る日との間の日数の見積もりを入力します。

プロジェクトの契約で個別のバッファー日を設定します。 日数は、売上請求書の期限と顧客の支払パターンに対する組織のエクスペリエンスの両方に基づいて計算されます。

#### <a name="actual-cash-inflow"></a>実際の現金収支

実際の現金収支は予測に似ていますが、最初の請求日から計算を開始できます。 次に例を示します。

-   **請求書の日付:** 2012 年 3 月 2 日。
-   **期限:** 2012 年 3 月 16 日。 支払条件は 14 日間に設定されています。
-   **予想売上支払期日:** 2012 年 3 月 29 日。 計算には、3 つの一般的なバッファー日と 10 の個別のバッファー日が含まれます。

#### <a name="cost-forecasting"></a>キャッシュ予測

定義された日数に基づいて、コスト支払期日はプロジェクトの日付と異なる場合があります。 この場合、コスト支払期日は、プロジェクトの日付からの日数を支払条件の日数に加算して計算されます。 

たとえば、トランザクションのプロジェクト日付は 2012 年 3 月 5 日であり、次の支払条件が設定されています。

-   **時間:** 当月 (**M**)
-   **経費:** 14 日間 (**D14**)
-   **品目:** 30 日 (**D30**)

これらの設定に基づいて、以下は各トランザクション タイプのコスト支払期日です。

-   **時間:** 選択した月の最終日である 2012 年 3 月 31 日。
-   **経費:** トランザクションから 14 日後の 2012 年 3 月 19 日。
-   **品目:** トランザクションの日付から 30 日後の 2012 年 4 月 4 日。

> [!NOTE] 
> 発注書の期限は、プロジェクトの発注書が作成されたときのベンダー トランザクションに基づいています。 期限はデフォルト設定では決定されていません。 

コスト支払期日は、バッファー日では計算されません。 プロジェクトが完了した後、すべての原価計算と請求が完了すると、コストと売上の両方が損益勘定に転記されます。 

すべての売上請求書とベンダー請求書が完了すると、**キャッシュ フロー** ページのフィールドと **プロジェクト ステートメント** ページのフィールド間の関係を表示できます。

### <a name="review-costs"></a>コストの確認

**コスト管理** ページ上で、プロジェクト中に組織により発生するコストを監視できます。 プロジェクトの元の予算原価と、現在の実際原価および確定済み費用を比較することで、プロジェクトが順調か、予算超過もしくは予算内かを判断できます。 

> [!NOTE] 
> **コスト管理** ページを使用してプロジェクト コストの現在のステータスを表示する場合、元の予算と予算残高に対して選択された予測モデルを使用します。 コストを計算するときに他の予測モデルを選択すると、計算結果は正確になりません。

#### <a name="viewing-the-remaining-budgeted-amounts"></a>予算残高額の表示

もし **予算残高** が **プロジェクト管理および会計パラメーター** ページでコスト管理メソッドとして選択される場合、**コスト管理** ページは、実績として転記されていないか、確定済みとしてマークされていないコストを計算します。 具体的には、**コスト管理** ページの下部ペインの **一般** タブの金額は次の方法で計算されます。

-   **実際のコスト** – 選択した費用明細行のプロジェクトに費やされた合計金額。 実際原価の金額は **元帳の更新** ページで計算されます。
-   **確定済み費用** – 法人が支払いを確定した追加の経費。 個々の確定済み費用は **確定済み費用ページ** で計算されます。
-   **予算残高** – 選択した費用明細行に対して引き続き使用できる元の予算金額。 予算残高は **総勘定元帳のプレビュー** ページで計算されます。
-   **合計コスト** – 実際のコスト、確定済みコスト、および予算残高の額の合計。

**コスト管理** ページの **偏差** タブで、予想総コストと元の予算の比較を表示できます。 この比較により、差額が表示されます。 これにより、データが一致しない箇所が分かります。 誤差は、次のようにして計算されます。

-   **元の予算** – 選択した費用明細行の元の予算金額。 元の予算金額は **総勘定元帳のプレビュー** ページで計算されます。
-   **原価合計** – **全般** タブに報告された、実際原価、実確定済み費用、および予算残高の合計。
-   **誤差** – 原価合計と元の予算の差額。
-   **数量に基づく差異** – 元の予測と合計予測の差額の合計。 この差異は、(合計予測数量) × (元の平均価格 – 合計平均価格) の数式で表すことができます。 この計算は、プロジェクト時間にのみ適用されます。
-   **価格に基づく差異** : 元の予測と合計予測の差額の合計。 この差は、(元の予測価格) × (元の予測数量 – 予測数量合計) として数学的に表すことができます。 この計算はプロジェクト時間にのみ適用されます。

#### <a name="viewing-the-total-budgeted-amounts"></a>予算額合計の表示

**プロジェクト管理と会計パラメーター** ページのコスト管理方法として **合計予算** が選択された場合、**コスト管理** ページでは、プロジェクトの実際のコストと総コストを計算して、両者の間の違いを検出できるようにします。 具体的には、**原価管理ページ** の **全般** タブにある下部ウィンドウの列で、金額が次のように計算されます。

-   **予算原価の合計** – 選択した費用明細行に対する元の合計予算額。
-   **実際原価** – 選択した費用明細行のプロジェクトでこれまでに発生した原価の合計。
-   **確定済み費用** – 選択した費用明細行に対して確定した金額の合計。
-   **差異** – 実際原価および確定済み費用の合計と原価合計の差額。 差異は、合計予算に対して追加の原価を割り当てる必要があるかどうかを示します。

**原価管理** ページの **誤差タブ** で、次のフィールドを見ることで総予算と元の予算との間の差額を確認できます。

-   **元の予算** – 費用明細行に対する元の予算金額。 元の予算は **総勘定元帳のプレビュー** ページで計算されます。
-   **予算原価の合計** – 費用明細行に対する元の予算原価の合計。 予算原価の合計は **総勘定元帳のプレビュー** ページで計算されます。
-   **誤差** – 費用明細行の誤差 この金額は、元の予算から原価合計を差し引くことで算出されます。
-   **数量に基づく差異** – 元の予算と合計予算の差額の合計。 この金額は、元の予算時間から合計予算時間を差し引き、その差に元の予算原価価格を乗算することで算出されます。 この差異は、(元の予算原価価格) × (元の予算時間 - 合計予算時間) の数式で表すことができます。 この計算は、プロジェクト時間にのみ適用されます。
-   **価格に基づく差異** – この金額は、元の予算時間から合計予算時間を差し引き、その差に消費時間の合計を乗算することで算出されます。 この差は、(合計消費時間) × (元の予算時間 - 合計予算時間) として数学的に表すことができます。 この計算はプロジェクト時間にのみ適用されます。

### <a name="analyze-utilization"></a>稼働率を分析する

稼働率とは、特定の作業期間に作業員が請求可能または生産的な作業を行う時間の割合です。 請求可能時間は、特定の顧客に請求できる作業者の時間です。 

作業者の稼働率は、請求可能な時間数を特定の期間の労働時間数で割って計算されます。 たとえば、ある期間の支払請求可能な時間が 30 時間で、その同じ期間の勤務時間が 40 時間であるならば、その作業者の稼働率は 75% になります。 

作業者の稼働率を計算する場合、支払請求可能率または実効率を計算できます。

-   **支払請求可能率** – 支払請求可能な時間と支払請求不可能な時間または基準時間との差。
-   **実効率** – 生産時間と非生産時間または基準時間との差。 生産時間は、特定のプロジェクトに寄与する作業者が費やす時間です。 内部プロジェクトの場合を除いて、通常、生産時間は顧客に請求されます。 非生産的な時間は決して顧客に請求されません。

**時間稼働率** ページで稼働率を計算します。 計算はデフォルトの設定に基づきます。 これらの設定では、各プロジェクトタイプに **稼働率** または **負担** を割り当てることで時間を計算する方法も指定します。 これは、支払請求可能なレートと実効率の計算に適用されます。

-   **稼働率** – 選択したプロジェクト タイプについて報告された時間は、常に支払請求可能な稼働または効率稼働が考慮されます。
-   **負担** – 選択したプロジェクト タイプに関して報告された時間は、常に支払請求不可能な稼働または非効率稼働が考慮されます。
-   **明細行プロパティに基づく** – 特定の時間トランザクションの明細行プロパティにより、時間が支払請求可能と見なされるか、効率稼働と見なされるかが決まります。
-   **含まれていない** – 時間は、請求対象または効率利用の計算に織り込まれません。

**時間稼働率** ページでは、作業者またはプロジェクトの全体的な稼働率のパーセンテージに加えて、次の各時間タイプの稼働率の計算に使用された時間数を表示できます。

-   **含まれない時間数** – この時間は時間使用率に算入されません。
-   **含まれる時間数** – 稼働時間と非稼働時間を加算して計算されます。 この時間は稼働率に算入されます。
-   **非稼働時間** – 支払請求可能なレートを計算する場合の請求不可能な時間と同じです。 実効率を計算する場合の非生産時間と同じです。
-   **稼働時間** – 支払請求可能なレートを計算する場合の請求可能な時間と同じです。 実効率を計算する場合の生産時間と同じです。

作業者の稼働率を計算する際には、基準時間または含まれる時間が使用できます。 含まれる時間を使用する場合は、入力された時間数の割合として計算されるため、作業者がタイムシートの期間の作業時間をすべて記録する必要があります。 プロジェクト、プロジェクト契約、顧客レコード、またはカテゴリの時間稼働率を計算するときは、計算に含まれる時間を使用する必要があります。

### <a name="review-project-statements"></a>プロジェクト ステートメントの確認

プロジェクト ステートメントを作成して、プロジェクトの進捗状況のクイック スナップショットを表示できます。 プロジェクト ステートメントを実行するとき、**全般** タブの **プロジェクト ステートメント** ページで選択することでステートメントの計算に使用される条件を指定できます。 次の情報を含めるか除外するかを選択できます。

-   プロジェクト タイプ
-   トランザクション タイプ
-   プロジェクト日付/元帳日付
-   データ

明細書が計算された後、**プロジェクト明細書** ページの各種タブで次の情報を表示できます。

-   **全般**: プロジェクトの基本的な損益構造に関する全般情報。
-   **利益と損失** – 未収売上に関する情報。
-   **WIP** – WIP 勘定残高に関する情報。
-   **消費** – 時間、アイテム、費用、および給与トランザクションの消費に関する情報。
-   **請求書**: 請求書および分割払請求に関する情報。
-   **時間率**: 売上勘定および原価勘定に転記される時間の時間率。
